# GitHub Pages 個人ホームページ計画

## 0. 現在の方針（2026-03-17 更新）

現在は次の方針で運用しています。

- サイトの主目的はスクリプト記録庫の運用
- デザイン比較用ページ（designs / concepts / 各preview）は廃止
- テーマは Copper File を既定として固定
- 日常更新は `Home` / `Scripts` / `About` と `_scripts/` を中心に行う

このため、以降の記述にある「まずMVPを作る」「公開先は最後に決める」という方針は維持しつつ、デザイン候補比較に関する作業は対象外として扱います。

## 1. このサイトの目的

このサイトは、個人ホームページとしての最小限の自己紹介を持ちつつ、主目的を「自分が使用したことのあるスクリプトの記録と公開」に置く。

目指す状態は次の通り。

- 自分が後から見ても再利用しやすい
- スクリプトの用途と使い方がすぐ分かる
- GitHub Pages 上で静的に安定公開できる
- 新しいスクリプトを追加しやすい
- 公開して問題ない内容だけを継続的に蓄積できる

## 2. Jekyll を前提にする理由

GitHub Pages でスクリプト集を公開する用途では、Jekyll は相性が良い。

- Markdown ベースで記事追加しやすい
- レイアウトを共通化しやすい
- カテゴリやタグで整理しやすい
- GitHub Pages と組み合わせやすい
- 静的サイトなので保守負荷が低い

今回のサイトは「プロフィールサイト」よりも「スクリプト記録庫」に近いため、Jekyll を基盤にする方が運用しやすい。

## 3. 最終ゴール

最終ゴールは、GitHub Pages 上で次を満たすサイトを公開すること。

1. トップページがある
2. スクリプト一覧ページがある
3. 各スクリプトの詳細ページがある
4. カテゴリ別に整理されている
5. GitHub リポジトリへの導線がある
6. 今後も Markdown ベースで継続追加できる

## 4. 公開前に決めること

公開作業を始める前に、次の内容を先に決めておく。

### 4.1 掲載対象

- 自分が実際に使ったスクリプト
- 2 回以上使った、または再利用価値があるもの
- 公開して問題ないもの
- 説明可能なもの

### 4.2 掲載しないもの

- API キーや認証情報を含むもの
- 業務固有情報や社内情報を含むもの
- ローカル環境に強く依存して再現困難なもの
- 第三者に誤用されやすい危険なもの

### 4.3 サイト構成

最低限、以下の構成を目標にする。

- トップページ
- スクリプト一覧ページ
- スクリプト詳細ページ
- 自己紹介ページ
- 更新履歴または新着欄

## 4.4 published の扱い

このサイトの `published` は、現状では「一覧に出すかどうか」を主目的にした運用項目です。

- `published: true`: 一覧に出す想定
- `published: false`: 一覧や導線から外す想定

ただし、これは強い非公開設定ではありません。
コレクション `_scripts/` は HTML 出力対象なので、`published: false` でもビルド結果に個別ページが含まれる可能性があります。

そのため、次の区別で運用するのが安全です。

- 下書き管理: `published: false`
- 本当に未出力にしたい原稿: 公開対象外ディレクトリで管理する、または公開用ブランチに載せない

## 5. 公開までのステップ

以下を、公開までの実施手順として採用する。

### Step 1. サイトの目的と運用方針を固定する

まず、サイトの目的を 1 文で固定する。

例:

> 自分が使用したスクリプトを、再利用しやすい形で記録し、必要に応じて公開するための個人サイト

この段階の完了条件:

- 何を載せるかが明確
- 何を載せないかが明確
- 主役が「スクリプト集」であると定義できている

### Step 2. スクリプト記事のテンプレートを決める

各スクリプトページで記録する項目を固定する。

推奨テンプレート:

1. タイトル
2. 概要
3. 使用言語
4. 想定環境
5. 入力
6. 出力
7. 実行方法
8. 注意点
9. コード
10. 更新日
11. GitHub リンク

この段階の完了条件:

- 新規追加時に迷わない
- ページ品質にばらつきが出にくい

### Step 3. Jekyll プロジェクトを作る

GitHub Pages 向けに Jekyll プロジェクトを作成する。

構成の例:

- `_config.yml`
- `_layouts/`
- `_includes/`
- `_posts/` または `_scripts/`
- `index.md`
- `about.md`
- `scripts.md`
- `assets/`

### Step 3 補足: 各構成に入るデータと目的

- `_config.yml`
	- 入るデータ: サイト全体の設定値（タイトル、URL、baseurl、collections、defaults、プラグイン設定など）
	- 目的: Jekyll の生成ルールを一元管理し、全ページに共通する動作を固定する
- `_layouts/`
	- 入るデータ: ページ骨組みのHTMLテンプレート（共通レイアウト、通常ページ用、スクリプト詳細用など）
	- 目的: 各ページの外枠を共通化して、見た目と構造の重複を減らす
- `_includes/`
	- 入るデータ: ヘッダー、フッター、head要素などの部分テンプレート
	- 目的: 小さな共通部品を再利用し、レイアウト修正を一箇所で反映できるようにする
- `_posts/` または `_scripts/`
	- 入るデータ: 記事本文のMarkdownと front matter（タイトル、要約、カテゴリ、公開フラグ、タグなど）
	- 目的: コンテンツ本体を蓄積する
	- 補足: このサイトではブログ標準の `_posts/` ではなく、スクリプト管理用の `_scripts/`（collection）を採用している
- `index.md`
	- 入るデータ: トップページの本文（サイトの目的、現在の運用方針、導線）
	- 目的: サイトの入口として全体像と主要リンクを提示する
- `about.md`
	- 入るデータ: サイトの方針、公開ルール、掲載対象・非対象の説明
	- 目的: 運用方針を明文化し、後から判断に迷わないようにする
- `scripts.md`
	- 入るデータ: `_scripts/` の一覧表示ロジックと一覧ページ本文
	- 目的: 蓄積したスクリプトを一箇所で参照できるようにする
- `assets/`
	- 入るデータ: CSS、画像などの静的ファイル
	- 目的: 見た目やメディアをページ本文と分離し、保守しやすくする

おすすめ方針:

- 通常のブログ形式に寄せるなら `_posts/`
- スクリプト専用の一覧性を重視するなら collection として `_scripts/` を使う

今回の用途では、後から整理しやすい collection ベースを優先してよい。

この段階の完了条件:

- Jekyll の基本構成がある
- ローカルでプレビューできる

### Step 4. MVP を作る

最初から全件移行しない。まずは代表的な 3 件だけ載せる。

選定基準:

- よく使ったもの
- 説明しやすいもの
- 公開に問題がないもの

MVP の内容:

- トップページ
- スクリプト一覧ページ
- スクリプト詳細ページ 3 件
- 簡単な自己紹介ページ

この段階の完了条件:

- 最低限の導線ができている
- サイトとして一通り閲覧できる

### Step 5. GitHub リポジトリを作成する

リポジトリは GitHub Pages を前提に作成する。

候補:

- ユーザーサイト: `username.github.io`
- プロジェクトサイト: 任意リポジトリ名

個人ホームページとして使うなら、ユーザーサイト形式の方が分かりやすい。

この段階の完了条件:

- Git リポジトリが作成済み
- リモートが設定済み
- 初回 push ができる

### Step 6. GitHub Pages を有効化する

GitHub 上で Pages を有効化する。

選択肢:

1. ブランチから公開
2. GitHub Actions で公開

最初はブランチ公開で十分。独自ビルドや制約回避が必要になったら Actions に移行する。

この段階の完了条件:

- 公開 URL が発行される
- トップページが表示される

### Step 7. 公開後の確認を行う

公開したら、次を確認する。

- リンク切れがない
- モバイルで読める
- コードブロックが崩れない
- 日本語表示に問題がない
- メタ情報やタイトルが意図通り

この段階の完了条件:

- 主要ページが問題なく閲覧できる
- 修正点が洗い出されている

### Step 8. 継続運用ルールを決める

サイトは公開より更新が重要。最低限、次をルール化する。

- 新規スクリプト追加時はテンプレート必須
- 公開前に秘匿情報を点検する
- 動作環境を必ず書く
- 月 1 回はリンクと内容を見直す

この段階の完了条件:

- 更新の手順が定まっている
- 放置しにくい状態になっている

## 7. 未出力に近づける設計案

`published` を今より強い意味で運用したい場合は、次の順で検討すると実装負荷が低いです。

1. 運用で分ける
	- 公開候補は `_scripts/`
	- 非公開の下書きは `drafts/` や `private/` のような Jekyll 出力対象外ディレクトリに置く
2. 公開用ブランチを分ける
	- 作業ブランチでは下書きを保持し、公開ブランチには公開対象だけを載せる
3. 将来的に自動化する
	- `published: true` の記事だけを公開用ディレクトリへコピーしてビルドする前処理を導入する

当面は 1 の運用分離が最も安全です。

Jekyll を使う場合、実装前に以下を整えておくと後で詰まりにくい。

### 6.1 Git と GitHub の基本設定

やっておくこと:

- Git をインストール
- GitHub アカウントを用意
- Git のユーザー名とメールを設定
- SSH または HTTPS の運用方針を決める

初期設定例:

```powershell
git config --global user.name "Your Name"
git config --global user.email "your-email@example.com"
git config --global init.defaultBranch main
```

### 6.2 Ruby と Bundler の準備

Jekyll は Ruby ベースなので、ローカルプレビュー用に Ruby 環境を整える。

Windows では RubyInstaller を使う構成が一般的。

やっておくこと:

- Ruby をインストール
- `ridk install` の推奨構成を適用
- Bundler を入れる

導入後の確認例:

```powershell
ruby -v
gem -v
bundle -v
```

Bundler 未導入なら:

```powershell
gem install bundler
```

### 6.3 Jekyll のローカル確認環境

ローカルで見えない状態で公開まで進めると手戻りが増える。最低限、ローカルプレビュー環境を作る。

一般的な流れ:

```powershell
bundle install
bundle exec jekyll serve
```

確認できること:

- レイアウト崩れ
- Markdown のレンダリング
- リンク切れ
- 日本語ページの見え方

### 6.4 VS Code の整備

最低限あるとよいもの:

- Markdown 編集支援
- YAML ハイライト
- Git 操作支援
- 任意で Ruby/Jekyll 関連補助

やっておくとよいこと:

- 改行コードを統一する
- 文字コードは UTF-8 を前提にする
- Markdown プレビューを使える状態にする

### 6.5 `.gitignore` の準備

Jekyll では生成物や依存関係を管理対象から外す。

代表例:

- `_site/`
- `.sass-cache/`
- `.jekyll-cache/`
- `.bundle/`
- `vendor/`

### 6.6 独自ドメインを使うかの判断

必要なら早めに決める。

- 使わないなら `username.github.io` で開始
- 使うなら DNS 設定と `CNAME` の管理が必要

最初は独自ドメインなしで開始し、後で追加でも問題ない。

### 6.7 画像とコードの置き方を先に決める

後から散らかりやすいので、配置ルールを決める。

例:

- 画像は `assets/images/`
- CSS は `assets/css/`
- スクリプト本文は collection 内
- 実ファイルを配布するなら `downloads/` などを分離

現時点では PNG などの静的画像用に [assets/images](assets/images) を作成済み。

### 6.8 公開前チェック項目をテンプレート化する

最低限のチェックリストを先に用意しておく。

- 秘匿情報が入っていないか
- 外部リンクが正しいか
- コードが読めるか
- 実行手順が書かれているか
- 環境依存の注意点があるか

Ruby と Bundler の準備ができたら、ルートディレクトリで次を実行する。

```powershell

```

ブラウザでは通常、次の URL で確認できる。

- `http://127.0.0.1:4000/`

## 11. 補足

GitHub Pages の標準 Jekyll ビルドは便利だが、使えるプラグインやビルド自由度に制約がある。最初はその制約内で始め、必要になった時点で GitHub Actions ベースのデプロイに切り替えるのが現実的。
