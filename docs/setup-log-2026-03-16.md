# Ruby と Bundler の導入作業ログ

## このドキュメントの目的

この文書は、2026-03-16 にこのワークスペースで実施した「Ruby と Bundler を入れて、Jekyll のローカル起動確認まで進める」作業を記録したものです。

「何をしたのか分からない」を避けるために、次の順でまとめています。

1. 何を目標にしたか
2. 実際に実行したこと
3. 途中で起きた問題
4. どう直したか
5. 最終的に何が確認できたか

## 目標

この作業の目標は次の3点でした。

1. Windows 上で Ruby を使えるようにする
2. Bundler で Jekyll の依存関係を入れられるようにする
3. このリポジトリの Jekyll サイトをローカルで起動して確認する

## 最初の状態

作業前は、次の状態でした。

- `ruby` コマンドが使えない
- `bundle` コマンドが使えない
- Jekyll サイトのひな形自体は作成済み
- ローカル起動は未確認

実際に確認した時点では、`ruby -v` と `bundle -v` はどちらも失敗していました。

## 実施した内容

### 1. インストール手段を確認した

まず、Windows で使えるパッケージマネージャーを確認しました。

実行した確認:

```powershell
winget --version
choco --version
git --version
```

結果:

- `winget` は利用可能
- `choco` は未導入
- `git` は利用可能

このため、Ruby の導入は `winget` を使う方針にしました。

### 2. RubyInstaller の候補を確認した

次に、`winget` で導入できる Ruby の候補を調べました。

実行した確認:

```powershell
winget search RubyInstaller --source winget --accept-source-agreements
```

確認できた候補の中から、Jekyll との相性と安定性を考えて `Ruby 3.2 with MSYS2` を選びました。

採用したパッケージID:

- `RubyInstallerTeam.RubyWithDevKit.3.2`

### 3. Ruby をインストールした

次のコマンドで Ruby を導入しました。

```powershell
winget install --id RubyInstallerTeam.RubyWithDevKit.3.2 --source winget --accept-source-agreements --accept-package-agreements --silent
```

結果:

- インストールは成功
- インストール先は `C:\Ruby32-x64`

### 4. Ruby / Gem / Bundler が使えるか確認した

導入直後は、既存ターミナルの PATH が更新されていなかったため、そのままでは `ruby` と `bundle` が認識されませんでした。

そのため、このセッションでは一時的に PATH を追加して確認しました。

実行したコマンド:

```powershell
$env:Path = 'C:\Ruby32-x64\bin;' + $env:Path
ruby -v
gem -v
bundle -v
```

確認結果:

- Ruby は利用可能
- RubyGems は利用可能
- Bundler は利用可能

確認できたバージョン:

- Ruby: `3.2.10`
- Gem: `3.4.19`
- Bundler: `2.4.19`

### 5. 今後のターミナルでも使えるように PATH を調整した

今後のシェルでも `ruby` を使いやすくするために、ユーザー PATH への追加も実施しました。

実行した内容:

- ユーザー環境変数 `Path` に `C:\Ruby32-x64\bin` を追加

補足:

- 既に開いているターミナルには反映されない場合があります
- 反映されない場合は、VS Code のターミナルを開き直すか VS Code 自体を再起動します

### 6. Jekyll の依存関係をインストールした

Ruby と Bundler が使えることを確認した後、このリポジトリの依存関係をインストールしました。

実行したコマンド:

```powershell
$env:Path = 'C:\Ruby32-x64\bin;' + $env:Path
bundle install
```

結果:

- 依存関係のインストールは成功
- `Gemfile.lock` が生成された
- native extension を含む gem もインストールできた

この時点で、Ruby のビルド系依存関係も問題なく動いていることが分かりました。

## 起きた問題と対応

### 問題 1. `ruby` と `bundle` が見つからない

症状:

- `ruby -v` が失敗
- `bundle -v` が失敗

原因:

- Ruby はインストールされたが、既存ターミナルの PATH に反映されていなかった

対応:

- 一時的に `$env:Path` へ `C:\Ruby32-x64\bin` を追加して確認
- ユーザー PATH にも追記

### 問題 2. Jekyll 起動時にタイムゾーンデータで失敗した

最初に Jekyll を起動したとき、次のエラーが発生しました。

- `TZInfo::DataSourceNotFound`
- `ZoneinfoDirectoryNotFound`

原因:

- Windows 環境では、Jekyll 実行時に `tzinfo-data` が必要になることがある

対応:

- [Gemfile](../Gemfile) に `tzinfo-data` を追加
- その後、再度 `bundle install` を実行

追加した内容:

```ruby
gem "tzinfo-data", platforms: %i[ mingw mswin x64_mingw jruby ]
```

結果:

- タイムゾーン関連エラーは解消
- Jekyll の起動が通る状態になった

## 最終確認

依存関係を入れ直したあと、Jekyll の開発サーバーを起動しました。

実行したコマンド:

```powershell
$env:Path = 'C:\Ruby32-x64\bin;' + $env:Path
bundle exec jekyll serve --host 127.0.0.1
```

その後、ローカル URL にアクセスしてページ内容を確認しました。

確認した URL:

- `http://127.0.0.1:4000/`

確認できたこと:

- トップページが応答した
- Jekyll の生成ページが返ってきた
- 作成済みの Home / Scripts / About の導線が見える状態だった

## この作業で変更されたもの

この作業により、ワークスペース側では主に次が変わりました。

- [Gemfile](../Gemfile): `tzinfo-data` を追加
- [Gemfile.lock](../Gemfile.lock): `bundle install` により生成
- [_site](../_site): ローカルビルドにより生成

補足:

- [_site](../_site) は生成物で、[.gitignore](../.gitignore) により Git 管理対象外です

## 今後この環境で使う基本コマンド

今後ローカル確認するときの基本コマンドはこれです。

```powershell
bundle install
bundle exec jekyll serve --host 127.0.0.1
```

もし新しいターミナルでも `ruby` や `bundle` が見つからない場合は、VS Code を再起動して PATH 反映を待ちます。

## まとめ

今回やったことを短くまとめると、次の通りです。

1. `winget` で RubyInstaller + DevKit を導入した
2. Ruby / Gem / Bundler が使えることを確認した
3. `bundle install` で Jekyll 依存関係を入れた
4. Windows 固有の `tzinfo-data` 問題を直した
5. `bundle exec jekyll serve` でローカル起動を確認した

この時点で、この Jekyll サイトは Windows 上でローカルプレビュー可能な状態です。
